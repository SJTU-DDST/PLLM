# PLLM HiberAgent 部署说明

## 1. 前置条件

- Linux + NVIDIA Driver；完整桌面感知使用 GNOME Shell 42。
- conda Python 3.12，不使用 uv。
- 可读的本地模型目录 `/mnt/ssd-storage/shared_models/NVIDIA-Nemotron-3-Super-120B-A12B-NVFP4`。
- CMake 3.20+ 与 libibverbs development headers 用于 RDMA bridge。
- vLLM 与 PLLM 控制端口只监听 loopback。

## 2. 环境与版本守卫

```bash
cd ~/PLLM
bash scripts/setup_conda.sh
conda activate pllm
python -c "import vllm, torch; print(vllm.__version__, torch.__version__)"
python scripts/apply_vllm_hibercache_patch.py --check
```

脚本固定 vLLM 0.25.1，使用 `pip --no-cache-dir`，并安装 editable PLLM。HiberCache patch 会检查 distribution version 和目标源码块；任一不匹配即失败，不会模糊修改未知版本。可用 `--revert` 恢复，首次安装保留 `.pllm.bak`。

## 3. SSD 目录

共享模型不复制、不修改。HiberCache 使用独立 20GB quota；Marlin runtime experts 使用另一个默认 80GiB tier：

```bash
sudo install -d -m 0750 -o "$USER" -g "$(id -gn)" /mnt/ssd-storage/pllm-cache
sudo install -d -m 0750 -o "$USER" -g "$(id -gn)" /mnt/ssd-storage/pllm-experts
```

检查：

```bash
test -r /mnt/ssd-storage/shared_models/NVIDIA-Nemotron-3-Super-120B-A12B-NVFP4/config.json
test -w /mnt/ssd-storage/pllm-cache
```

## 4. 配置

首次启动生成 `~/.config/pllm/config.toml`，数据写入 `~/.local/share/pllm/events.sqlite3`。重要字段：

- `mode`：`auto`、`ai_priority`、`foreground_priority`、`keep_sleeping`。
- `yield_resume_idle_seconds` / `resume_idle_seconds`：Level 0 与深度休眠的恢复迟滞。
- `hibercache_dir` / `hibercache_staging_mb` / `hibercache_quota_gb`。
- `expert_slots_per_layer` / `expert_runtime_cache_dir` / `expert_runtime_cache_quota_gib`。
- `expert_auto_resize_enabled` / `expert_resize_cooldown_seconds`。

`expert_auto_resize_enabled` 默认是 `false`。在完整 runtime export、40 层 slot correctness 和真实 route calibration 验收前，只允许通过控制中心或 API 显式应用物理 profile。
- `expert_runtime_socket` / `expert_rdma_port` / `expert_rdma_binary`。
- `loader_mode`：`auto`、`fastsafetensors` 或 fallback。
- `rdma_peer`：远端暖源；为空时只走 NVMe。
- `remote_weight_source_enabled`：只有 ModelExpress/NIXL 暖源验收后才设为 true。
- `game_patterns` / `creative_patterns` / `excluded_process_patterns`。
- `dry_run`：只决策，不控制服务。

## 7. RDMA bridge

```bash
cmake -S rdma_bridge -B rdma_bridge/build -DCMAKE_BUILD_TYPE=Release
cmake --build rdma_bridge/build -j
python scripts/rdma_benchmark.py --allocator aligned --device mlx5_0
```

GPU 空闲后才使用 `--allocator cuda-host`。双机时对端：

```bash
python scripts/rdma_benchmark.py --server --device mlx5_0
```

本机：

```bash
python scripts/rdma_benchmark.py --peer 192.168.70.71 --device mlx5_0
```

DGX Spark 必须显示 host-staged；若 capability probe 发现 GB10 却选择 GPU Direct，应停止实验。

## 8. 真实 vLLM

确认 GPU 空闲后：

```bash
bash scripts/run_vllm.sh
```

启动参数包括：

- `VLLM_SERVER_DEV_MODE=1`、`--enable-sleep-mode`。
- NVFP4/Marlin、FP8 KV、32K、2 并发、0.85 内存上限。
- `--load-format fastsafetensors`。
- `OffloadingConnector + TieringOffloadingSpec`。
- `VLLM_FASTSAFETENSORS_QUEUE_SIZE=0`。

随后：

```bash
bash scripts/run_daemon.sh
bash scripts/run_desktop.sh
```

浏览器控制中心为 `http://127.0.0.1:17860`，OpenAI base URL 为 `http://127.0.0.1:17860/v1`。

### 8.1 Expert runtime cache 与弹性 vLLM

以下命令全部等待 GPU 空闲后执行。第一次完整导出 transformed Marlin rows：

```bash
bash scripts/run_vllm_export_experts.sh
python scripts/eer_runtime_ctl.py status
```

确认 `runtime-manifest.json` complete 后，以 128 slots/layer 启动：

```bash
PLLM_EER_SLOTS_PER_LAYER=128 bash scripts/run_vllm_eer.sh
```

远端 warm source：

```bash
install -d -m 0700 ~/.config/pllm
openssl rand -hex 32 > ~/.config/pllm/rdma-token
chmod 0600 ~/.config/pllm/rdma-token
systemctl --user enable --now pllm-rdma-store pllm-rdma-state
PLLM_EER_RDMA_PEER=192.168.70.71 \
PLLM_EER_RDMA_TOKEN_FILE=~/.config/pllm/rdma-token \
bash scripts/run_vllm_eer.sh
```

`pllm-rdma-store` 的 17900 根目录保存 immutable expert objects；`pllm-rdma-state` 的 17901 根目录保存 live-state transactions。两者共用 token 但不共用 object namespace。启用 state remote tier 时在配置中设置 `hiberstate_rdma_enabled=true`、`hiberstate_rdma_peer` 和相关端口/路径。

详细格式、故障语义和验收见 `docs/数据面实现与待验收说明.md`。

## 9. systemd user service

```bash
bash scripts/install_user_services.sh
systemctl --user start pllm-daemon pllm-desktop
journalctl --user -u pllm-daemon -f
```

模型服务不默认随登录启动，避免意外占满 GPU。

## 10. 故障与降级

- `controllable=false`：检查 vLLM dev mode 与 sleep mode；PLLM 不会 kill 该进程。
- HiberCache disabled：检查 `/mnt/ssd-storage/pllm-cache` 所有权与剩余空间。
- patch missing：Level 0 仍可演示；Level 2 连续性退回 token 重算/客户端重试。
- NIXL/ModelExpress missing：使用本地 NVMe，不影响核心闭环。
- GB10 GDR false：这是预期，使用 `cudaHostAlloc + ibv_reg_mr`。
- Level 1 主存不足：切 Level 2；UMA 默认就是 Level 2。
- 真实 Level 2 exact match 失败：正式演示固定 Level 0，无损声明降级为已验证范围。
- EER runtime socket missing：普通 vLLM 继续可用，但 expert plan 保持不可执行。
- runtime manifest incomplete：禁止 elastic 启动，重新完成全量导出或配置 RDMA warm source。
- resize 失败：服务保持 Level-0 quiesced；重试同一 profile 或重启，不手工 wake。
