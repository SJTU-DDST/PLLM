# DGX Spark 黑客松十日谈：让后台 AI 学会暂停，也学会回来

第一天的问题很生活化：桌面终于放得下 100B 模型，但用户一打开 Blender，为什么还要自己去终端关服务？云端调度默认 GPU 属于推理，桌面却恰好相反，前台的人应该拥有最终优先级。PLLM 因此不是一个“显存清理按钮”，而是一位知道何时让路的 HiberAgent。

我们先做了最克制的版本。它只兼容 vLLM，只控制能通过 Sleep API 证明身份的服务，训练程序和未知 CUDA 进程一律不能碰。开发机恰好有一项长期训练占了约 86GB 显存，这反而成了很好的安全测试：PLLM 看见真实 99% SM 压力，但只让 mock vLLM 休眠，训练 PID 没有变化。

随后问题变难了。旧方案用 `abort + 手工重放`，能释放资源，却不能说“可恢复推理”。我们沿着 FastServe 的 token 抢占、Sereno 的前台 QoS、ServerlessLLM 的多级加载和 MAIO 的 page-cache 思路，把状态机改成 `ACTIVE -> YIELDING -> QUIESCING -> HIBERNATED -> RESTORING`。Level 0 使用 `mode=keep`，mock 测试第一次在回答中间停住：暂停期间一个 chunk 都没有多出来，唤醒后句子完整接上。

HiberCache 是最需要诚实的部分。Attention KV 可以交给 vLLM OffloadingConnector 和 SSD tier，但 NemotronH 还有 recurrent/Mamba state。我们没有为了讲故事发明一个“已经保存”的状态文件，而是把当前策略写清：已有 KV block 命中，缺失尾块和混合状态按 token 重算；真实模型贪心输出必须与不中断基线哈希一致，否则 Demo 就回退 Level 0。

DGX Spark 又带来一个反直觉事实：它是 UMA，却不支持 GPUDirect RDMA。于是远端暖源不是漂亮的一箭直达 GPU，而是 ConnectX-7 到 `cudaHostAlloc` MR，再进入 UMA loader。我们写了独立 C++20 bridge，在当前 mlx5_0 上真实完成 PD/MR 注册；131.204Gb/s 只叫 host staging bandwidth，不冒充 100Gbps 网络吞吐。

界面最后才开始做，却帮助我们重新审视系统。四级内存流必须说清数据在哪里，成本条必须解释为什么选 Level 2，token N 必须证明请求不是重新提问。评委模式也有醒目的历史回放标签。一个好的演示不是把所有灯点绿，而是让人知道哪些能力已验证、哪些会回退、哪些数字要等 GPU 空闲。

PLLM 想表达的最终观点很简单：桌面 AI 的能力不只由“能加载多大的模型”定义，也由它能否与人的工作和平共处定义。会生成代码只是智能；知道暂停，并带着原任务回来，才是桌面运行时应有的礼貌。
