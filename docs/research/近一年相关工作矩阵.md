# 近一年 LLM 推理休眠、状态恢复与多级装载研究矩阵

## 1. 调研口径

本调研以 2026-07-17 为截止日，核心时间窗为 2025-07-17 至 2026-07-17。只用论文主页、会议论文、arXiv 原文、开源框架源码与 NVIDIA 官方文档作为事实来源。ServerlessLLM、Mooncake、NEO 等早于该窗口但构成直接技术基线的工作单列为“基础基线”，不冒充近一年成果。

目标问题不是一般的“如何让 LLM 更快”，而是以下五个条件能否同时成立：

1. 前台应用提出资源请求后，在有限 deadline 内停止后台推理并释放物理内存；
2. 不终止用户正在生成的请求，恢复后保持同一 token 流的逻辑连续性；
3. 支持 Attention + recurrent/SSM + MoE 的混合模型状态，而非只处理 Transformer KV；
4. 在 DGX Spark 的 128GB UMA、NVMe 与不支持 GPUDirect RDMA 的 ConnectX-7 路径上成立。
5. 中等压力下保持原始 MoE Top-k 路由，动态释放 routed-expert 物理容量；paging 不可行时平滑转入事务式休眠。

## 2. 一页结论

近一年工作已经分别解决了四个子问题，但尚未形成上述闭环：

- **让计算让路**：Sereno 用细粒度 yield point 保护移动前台 QoS；FastServe 与 ConServe 支持 token 或 layer 粒度抢占。
- **让内存让路**：KunServe 丢弃跨副本冗余参数，Prism 用 GPU memory ballooning 弹性回收多模型内存。
- **保存可变状态**：Concordia 用持久 kernel 做增量 checkpoint，GhostServe 用 host parity 保护 KV，Sparse Prefix Caching 为 recurrent state 提供稀疏精确 checkpoint。
- **从慢层恢复**：MAIO 优化 page cache，SolidAttention、Tutti、LMCache 和 KVDrive 优化 SSD/网络 KV，FaaScale、ModelExpress 和 ServerlessLLM 优化模型装载。

关键空缺被进一步收窄为：现有工作分别研究固定预算的 expert offloading、前台 compute yield、live-state checkpoint 和模型装载，但较少研究前台物理容量、I/O 与 compute envelope 同时变化时，exact expert residency 何时仍可行、何时必须把混合 live request 转为事务式休眠。这个缺口仍需通过组合基线和真实实验验证，不能仅凭文献中没有同名系统就宣称新颖。

## 5. vLLM 0.25.1 的真实能力边界

源码核对结果如下：

- `mode=keep` 将 scheduler 置为 `PAUSED_ALL`，当前 step 完成后不再调度，并保留 request object 与原 KV allocation。
- Level 1 把 `weights` tag 复制到 CPU，丢弃其他 allocation；Level 2 丢弃权重和 KV。官方文档也明确说明两级都会忘记 KV 内容。
- 在离散 GPU 上，Level 1 的 CPU backup 是独立容量层；在 DGX Spark UMA 上，它仍占同一 128GB 物理内存，且可能产生第二份 74.8GiB 权重，因此不是“释放内存”。
- `OffloadingConnector` 已识别 `MambaSpec`，但 connector 的 prefix reuse 语义不等于 live request transaction。深度 sleep 后仍需证明 scheduler block table、Mamba state、sampling state 与 stream output 同属于一个 commit epoch。

因此，`keep + external KV connector` 是实现载体，不是无损深度休眠已经被 vLLM 解决的证据。

## 6. NemotronH 状态定量分析

本地只读模型 `NVIDIA-Nemotron-3-Super-120B-A12B-NVFP4` 的 17 个 safetensors shard 合计 `80,317,948,856` bytes，即 `74.802GiB`。配置包含 88 层：40 个 Mamba 层、40 个 MoE 层和 8 个 Attention 层。

单请求、TP=1、32K context 的状态下界为：

| 对象 | 公式 | 大小 |
| --- | --- | ---: |
| Mamba temporal state | `40 * 128 heads * 64 * 128 * FP32` | 160.000MiB |
| Mamba conv state | `40 * (8192 + 2*8*128) * 3 * BF16` | 2.344MiB |
| Attention KV | `8 * 32768 * 2(K,V) * 2 KV heads * 128 * FP8` | 128.000MiB |
| 合计下界 | 不含 allocator padding、grammar 与 block table | 290.344MiB |

这带来三个系统含义：

1. 权重比单请求可变状态大约 264 倍。休眠时写出第二份权重是高写放大且无必要的；应直接丢弃权重，恢复时读取现有 canonical shard。
2. Attention KV 是 append-only block，适合增量 shadow checkpoint；Mamba recurrent state 每个 token 都整体演化，dirty-page delta 不会天然稀疏。
3. 对 Mamba 最合理的选择是“复制当前 162MiB state”与“从较旧 checkpoint teacher-force 重放少量 token”二选一，而不是假设所有状态都按 KV block 管理。

## 7. DGX Spark 约束

[NVIDIA 官方规格](https://docs.nvidia.com/dgx/dgx-spark/hardware.html)给出 128GB LPDDR5x UMA、273GB/s 内存带宽、1TB/4TB NVMe、两个 copy engine、140W GB10 TDP；[产品规格页](https://www.nvidia.com/en-us/products/workstations/dgx-spark/)标注 ConnectX-7 为 200Gbps。官方移植指南还明确指出：

- CPU 与 iGPU 动态共享同一物理内存，没有固定 VRAM carve-out；
- `cudaMemGetInfo` 不包含 OS 可回收 page cache/swap，资源判断应结合 `MemAvailable` 与 PSI；
- DGX Spark 不支持 GPUDirect RDMA、nvidia-peermem、DMA-BUF 或 GDRCopy；verbs 应回退到 `cudaHostAlloc` 后 `ibv_reg_mr` 的 host-staged path。

由此，Spark 上有效层级不是传统的 `HBM -> CPU DRAM -> SSD`，而是：

```text
GPU allocation / CPU pages (同一 128GB UMA)
                |
                +-> local NVMe --O_DIRECT/io_uring--> pinned staging
                |
                +-> remote host --CX-7 RDMA---------> pinned staging
                                                       |
                                                       +-> CUDA copy/model tensor
```

CPU memory只能作为短暂 staging，不能作为深度休眠的容量层。

## 8. 可发表的研究缺口

一个足够明确且可证伪的新问题是：

> 如何在单机 UMA AI PC 上，于前台 deadline 内释放一个正在生成的混合 LLM 实例，同时保持同一 token 流可精确恢复，并最小化从本地 NVMe 或无 GDR 的远端暖源恢复所需时间？

预期贡献不应写成“首次使用 SSD/RDMA”或“首次 checkpoint LLM”，这些声明会被现有工作直接推翻。更可信的贡献边界是：

1. **架构感知的事务式休眠协议**：统一 Attention KV、Mamba recurrent state、sampler/grammar、scheduler mapping 和客户端 token ledger；把 immutable weights 排除在 checkpoint 外。
2. **deadline 约束的 copy-or-replay planner**：针对 append-only KV 与全量更新 SSM 选择不同 checkpoint/recompute frontier，并把前台风险、恢复 horizon、能耗和 SSD 写放大放在同一优化问题中。
3. **面向 UMA/no-GDR 的双源恢复路径**：本地 direct NVMe 与 ConnectX-7 host-staged RDMA 分块协同，不沿用数据中心 GDR 假设，并显式计入共享内存带宽对前台的干扰。
4. **逻辑 exactly-once token stream**：保存已输出 token ID 与序号；从较旧状态恢复时 teacher-force 重放已提交 token，代理抑制重复输出。标准 OpenAI SSE 无 ACK，因此跨连接 exactly-once 仅对支持 sequence ID 的 PLLM 客户端成立。

## 9. 论文设计的否证条件

以下任一结果都应视为核心假设被削弱，而不是通过改口径掩盖：

- 290MiB 混合状态无法在前台 deadline 内持久化，且重算也无法满足恢复 SLO；
- `cuMemUnmap` 后 `MemAvailable` 与前台可用吞吐没有显著改善；
- host-staged RDMA 与 NVMe 并发导致 UMA 干扰，使双源恢复慢于最佳单源；
- teacher-force 重放后 greedy token 序列与 uninterrupted baseline 不一致；
- 休眠 checkpoint 写放大抵消了资源释放带来的能耗收益；
- 前台启动检测晚于其大规模内存分配，导致 release deadline 本身没有用户价值。

这些条件将直接映射到后续实验，不在当前 GPU 忙碌阶段预填结果。

## 10. 动态专家驻留相关工作补充

最新方案固定 Nemotron 原始 Top-22，不研究动态 Top-k 或 cache-aware rerouting。相关工作矩阵如下：

| 工作 | 核心机制 | 是否保持原路由 | 与 HiberFlow-EER 的差异 |
| --- | --- | --- | --- |
| [MoE-Infinity](https://arxiv.org/abs/2401.14361) | batch=1 稀疏 activation trace、cache replacement 与 prefetch | 是，miss 加载 | 固定设备预算，不感知外部前台 deadline/UMA capacity |
| [ProMoE](https://arxiv.org/abs/2410.22134) | 用中间结果预测后续 expert 并 proactive cache | 主要为精确预取 | 目标是隐藏 CPU-GPU transfer，不联合 hibernation phase boundary |
| [ExpertFlow](https://arxiv.org/abs/2410.17954) | routing-path predictor、token scheduling、predictive cache | 有 runtime correction | 优化模型吞吐；不建模桌面前台物理容量与 I/O SLO |
| [Fate](https://arxiv.org/abs/2502.12224) | cross-layer gate prediction、浅层优先 cache | 是，预测用于预取 | 依赖独立 CPU/GPU 层级，不处理 Spark UMA/no-GDR |
| [Pre-Attention Prediction](https://arxiv.org/abs/2511.10676) | 同层 attention 前表示和轻量 ranking predictor | 是，预测用于预取 | 提供 predictor，不解决动态 resident budget 与 tier selection |
| [SpecMD](https://machinelearning.apple.com/research/specmd-expert-prefetching) | 系统比较 cache policy，Least-Stale eviction | 是 | 揭示 LRU/LFU 假设不足；没有外部前台 resource envelope |
| [ActiveEvict](https://openreview.net/pdf?id=UAMZ4tRFn6) | pre-eviction 与动态 effective budget | 论文含 budget-aware routing，报告 <1% quality loss | 最接近动态 budget；PLLM 必须以 exact route、前台 SLO 与 hibernation 组合基线证明额外价值 |
| [FlashMoE](https://arxiv.org/abs/2601.17063) | SSD expert cache 与 ML replacement | 是 | 最接近 SSD tier；PLLM 额外受 UMA/前台 NVMe contention 约束 |
| [OD-MoE](https://arxiv.org/abs/2512.03927) | 多边缘节点按需加载，数层前 prediction | 是，JIT 加载 | 分布式 edge throughput，不保存 live request transaction |
| [SSD Energy Analysis](https://arxiv.org/abs/2508.06978) | 定量分析 decode expert SSD reads 的能耗 | 不适用 | 说明 prefetch 可隐藏延迟但不能消除能耗，构成 PLLM I/O governor 的反例基线 |

### 10.1 不能再使用的创新声明

- 首次预测 MoE experts；
- 首次动态换入换出 experts；
- 首次用 SSD/RDMA 保存 experts；
- 首次使用动态 expert cache budget；
- expert cache shrink 自动释放 GPU 算力。

### 10.2 可检验的新主张

1. 前台 envelope 同时改变容量、I/O、compute 和 deadline，产生可测的 elastic-to-hibernate phase boundary。
2. Prediction set 仅决定预取，不改变 actual Top-22；在相同物理 resident bytes 下，风险校准集合比 fixed top-n 降低 p95 blocking miss。
3. 在 UMA 上，CPU offload 不是容量层；NVMe/RDMA expert traffic 必须与前台共享带宽和功耗 governor。
4. 当 paging 不可行时，同一 live Attention/Mamba/token stream 可以事务式进入深度休眠，而不是 abort/replay 整个请求。

这四项只有在 slot data plane、组合基线和真实前台实验完成后才能从“研究假设”升级为“贡献”。
