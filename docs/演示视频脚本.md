# PLLM HiberFlow-EER 演示视频脚本

目标时长 2 分 40 秒。所有角标使用 `LIVE RTX PRO`、`MOCK-VALIDATED`、`HISTORICAL REPLAY` 或 `DGX SPARK PENDING`。

## 0:00-0:18 问题

画面：Nemotron 流式生成 CUDA kernel，右侧 `nvidia-smi` 显示模型常驻。启动 Blender render 前停顿一秒。

旁白：DGX Spark 可以把 100B 模型放在桌面，但桌面 GPU 还属于正在创作和游戏的用户。传统 vLLM 不知道前台任务什么时候需要资源。

## 0:18-0:42 前台感知

画面：打开实时 Vue 控制中心。EER 主指标显示物理数据面 `128/512`，决策行单独显示 planner `PLAN 256`；切换到评委模式回放 Blender 轨迹，页头明确显示 `HISTORICAL REPLAY`。

旁白：PLLM 不只看总利用率。它读取 GNOME 焦点、进程级 GPU、内存压力和供电状态。实时物理槽位与规划值严格分栏，历史 Blender 轨迹不冒充本机正在运行的 workload。

## 0:42-1:08 快速释放

画面：点击“释放资源”，状态进入 `QUIESCING -> HIBERNATED`；并排展示实测 JSON 与 `nvidia-smi`：约 46.1GiB 降至 1.31GiB，动作 184.5ms。

旁白：Level 2 直接丢弃已有只读 checkpoint 的不可变权重，不重复写出 75GB。PLLM 这次回收 44.17GiB；恢复会重新读取共享 shard，因此释放快不等于恢复快。

## 1:08-1:28 前台 QoS

画面：运行 `benchmark_foreground_admission.py`。常驻态申请 60GiB 显示 CUDA OOM；休眠态同一申请 2.52ms 成功。

旁白：这是前台显存 admission 的真实证据，不是 Blender 帧率。游戏、渲染与编码的 90% throughput 目标仍标记为待测，绝不拿 allocation 替代。

## 1:28-1:55 同流恢复

画面：状态进入 `RESTORING`，17 个 safetensors shard 进度倍速播放；41.39s 后 EER 重新 ready，代理请求返回 HTTP 200。旁边展示 Level 0 同一 stream 暂停期间 `chunks=0`。

旁白：Level 0 已证明同一连接冻结后继续，不需要重连；但跨独立请求输出不是 bitwise deterministic，所以当前不声称深度休眠 exact token resume。恢复后的第一个短请求仍约 33 秒，这是下一步必须解决的瓶颈。

## 1:55-2:18 DGX Spark 适配

画面：展示 `rdma_store_live.json`：20MiB RC PUT/GET、三份 SHA-256 一致；随后展示 `cudaHostAlloc` MR staging 结果，并将“network bandwidth 未测”放在同一屏。

旁白：当前证明的是 host-staged RDMA 数据语义，不是独立双机吞吐。DGX Spark 不支持 GPUDirect RDMA，目标路径使用 cudaHostAlloc 注册 MR；网络和 host-copy 指标始终分开。

## 2:18-2:40 工程与结尾

画面：展示 `55 passed`、CMake build、20,480-object manifest、实时桌面与移动控制中心；最后停在验收矩阵，PASS 与 NOT RUN 同屏。

旁白：PLLM 已经能让 120B 模型在用户需要 GPU 时快速让路，但 128-slot fused batch 上限、约 40 秒恢复、真实前台吞吐和 DGX Spark UMA 仍是公开问题。我们展示系统能做什么，也展示下一项研究究竟难在哪里。

## 无 GPU 备用演示

若比赛环境不能加载模型，点击“评委模式”运行历史轨迹。页头必须保留 `HISTORICAL REPLAY`，同时展示本轮保存的真实 JSON、截图和 HCA 结果；不把历史释放数字伪装成现场测量。
