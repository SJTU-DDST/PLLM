# PLLM Expert 数据面实现与验收说明

> 状态：`LIVE VALIDATED ON RTX PRO / DGX SPARK PENDING`。本轮已执行 pytest、CMake、vLLM、CUDA allocation、完整 runtime export、durable RC object path 和 75→71 volatile-pool v1；v2 跨机复测、RDMA-to-slot、真实 route predictor、Mamba serializer 与 DGX Spark UMA 仍待验收。

## 1. 数据对象

`ExpertCatalog` 现在为每个 checkpoint tensor 记录 shard、dtype、shape、safetensors data offset 和绝对 file offset。`SafetensorsExpertSource` 使用 `os.pread` 只读取指定 expert 的 weight/scales，不 mmap 或反序列化整个 shard。

Marlin 执行布局与 checkpoint 布局不同，因此实时换入使用一次性导出的 runtime object：

```text
.pllmex fixed header
  -> JSON: model fingerprint / layer / expert / dtype / shape / offsets / SHA-256
  -> Marlin w13/w2 weights and scales
```

SSD 写入采用 temporary file、`fsync`、atomic rename 和 parent-directory `fsync`。对象只有在 model fingerprint、format、tensor layout 和 SHA-256 全部通过后才可进入 physical slot。默认目录 `/mnt/ssd-storage/$USER/pllm-experts`，默认 quota 80GiB。

## 2. vLLM 物理槽位

`vllm_patch/sitecustomize.py` 仅在 `PLLM_EER_MODE=export|elastic` 时安装 runtime hook，并拒绝非 vLLM 0.25.1。

弹性模式固定要求：

- `NVIDIA-Nemotron-3-Super-120B-A12B-NVFP4`；
- ModelOpt NVFP4；
- Marlin modular MoE backend；
- `--enforce-eager`；
- 每层 slots 不少于原始 Top-22。

模型构造时把 routed weight 第一维从 512 改为配置的 physical slots，但保留 512-entry global expert map。物理 rows 先确定性清零，让 Marlin 建立目标形状；弹性启动强制使用 lazy `safetensors` loader，并在 `get_tensor` 前跳过全部 numeric routed-expert weight/scales，随后从 runtime cache 覆盖每个初始 slot。因此 checkpoint 冷启动只读取非 routed 权重，routed bytes 直接来自 SSD/RDMA runtime objects。每次 Marlin `apply` 前，runtime 从 actual `topk_ids` 得到必须存在的 expert 集合；命中直接使用，miss 同步加载并校验。只有所有写入完成后才把 logical ID 发布到 GPU `expert_map`。预测器不参与正确性路径。

LRU eviction 先撤销 mapping，再覆盖 slot。旧 bytes 不清零，因为不可达 bytes 不影响推理且清零不能释放 allocation。真正的容量变化使用 quiesced resize：删除旧 layer parameters、清理 allocator、创建更小 tensors、恢复 retained experts并重建 Marlin kernel。

## 3. SSD 导出

第一次使用必须在 GPU 空闲时完整加载一次模型：

```bash
conda activate pllm
bash scripts/run_vllm_export_experts.sh
python scripts/eer_runtime_ctl.py status
```

导出完成条件是 `runtime-manifest.json` 中：

```json
{
  "complete": true,
  "format": "vllm_runtime_nvfp4_marlin_v1",
  "objects": 20480
}
```

该步骤预计产生约 routed runtime weight 规模的额外 SSD 数据，实际大小必须以导出后 `du` 为准。quota 无法容纳全部对象时不会提交 complete manifest。

随后启动弹性服务：

```bash
PLLM_EER_SLOTS_PER_LAYER=128 bash scripts/run_vllm_eer.sh
bash scripts/run_daemon.sh
```

## 4. RDMA 数据路径

`pllm-rdma-store` 实现真正的 object get/put，不调用 benchmark 代替：

1. TCP 控制面交换 operation、object key、size、RC QP 和 MR；
2. sender 分块执行 `IBV_WR_RDMA_WRITE`；
3. PUT 服务端原子写入远端 SSD 后返回 commit ACK；
4. Python 重新验证 `.pllmex` SHA-256 与 model fingerprint；
5. 独立 bridge 进程把 registered host buffer 原子写入本地 SSD cache；Python 再校验并复制到 CUDA/UMA slot。

远端服务：

```bash
install -d -m 0700 ~/.config/pllm
openssl rand -hex 32 > ~/.config/pllm/rdma-token
chmod 0600 ~/.config/pllm/rdma-token
cmake -S rdma_bridge -B rdma_bridge/build -DCMAKE_BUILD_TYPE=Release
cmake --build rdma_bridge/build -j
rdma_bridge/build/pllm-rdma-store \
  --server /mnt/ssd-storage/$USER/pllm-experts --port 17900 \
  --allocator cuda-host --device mlx5_0 \
  --token-file ~/.config/pllm/rdma-token
```

同步本地 warm store：

```bash
python scripts/sync_expert_store_rdma.py \
  --peer 192.168.70.71 --port 17900 \
  --token-file ~/.config/pllm/rdma-token
```

全量同步先校验现有本地 package，再直接 PUT，不为上传重复写约 59GiB 本地数据；提交顺序为所有 `.pllmex` objects 在前、`runtime-manifest.json` 在最后。使用 `--start` 或 `--limit` 的部分同步不会上传 manifest。Elastic 计算节点若本地没有 complete manifest，会先从 RDMA peer 拉取并验证 schema、vLLM 0.25.1、runtime format、model fingerprint 与至少 20,480 objects；验证失败时拒绝启动，而不是等到 decode miss 才发现远端不完整。

计算节点使用：

```bash
PLLM_EER_RDMA_PEER=192.168.70.71 \
PLLM_EER_SLOTS_PER_LAYER=128 \
bash scripts/run_vllm_eer.sh
```

当前 DGX Spark 路径是 `ConnectX-7 -> bridge registered host buffer -> local SSD cache -> Python host buffer -> CUDA/UMA slot`，不声明 GPUDirect RDMA。它比理想的 in-process pinned-buffer 路径多一次本地写入和读取，实验必须单独报告这部分放大。

Elastic runtime 和 live-state fetch 默认显式传递 `--allocator cuda-host`；纯 CPU warm-source 同步工具默认 `--allocator aligned`，可按主机能力覆盖。二者都注册 host MR，不产生 GDR 声明。

RoCE 的 `ib_port` 与 `gid_index` 也可显式配置。当前测试主机 `192.168.70.75` 对应 `mlx5_0/1` 的 GID index 5；部署到 DGX Spark 时必须以 `show_gids` 的实际输出为准，不能硬编码该数字。

Token 在 TCP 控制通道中传输，不提供链路加密；该服务只适合受信任的专用 RDMA fabric，并仍需主机防火墙限制 17900/TCP 和 17901/TCP。无 token 只能通过显式 `--insecure-no-auth` 启动。Live state 可能含用户 prompt 与输出，当前也未提供传输或静态加密，不能部署在不可信网络或共享存储。

## 5. Live-state 事务承载层

`pllm/hiberstate.py` 为 Attention KV、Mamba recurrent state、sampler/RNG、block table 和 output ledger 提供通用 byte carrier。每个 committed epoch：

1. 按默认 64MiB 切分组件，单块不超过 RDMA bridge 的 128MiB 上限；
2. 对每块和完整组件分别记录 SHA-256、dtype、shape 与组件 metadata；
3. 先 durable 写所有 chunks，最后写 `commit.json`，再原子 rename 整个 transaction directory；
4. 远端复制先 PUT chunks，最后 PUT manifest；fetch 先取 manifest，再逐块拉取并在本地提交前完整校验；
5. quota/LRU 以整个 epoch 为单位删除，不逐块破坏已提交事务。

本地目录为 `/mnt/ssd-storage/pllm-cache/live-state`。远端状态服务使用独立的 `17901/TCP` 和 `pllm-rdma-state.service`，避免与 17900 上的 immutable expert store 混用。`scripts/hiberstate_ctl.py` 可执行 `status`、`replicate` 和 `fetch`。

该层当前 `serializer_attached=false`：它能可靠承载调用者给出的 state bytes，但尚未接入 vLLM 0.25.1/NemotronH 内部 Mamba state、KV block table 与 RNG serializer，因此不能据此声称真实请求已经可 exact-resume。

## 6. 控制与故障语义

vLLM 进程通过 `$XDG_RUNTIME_DIR/pllm-eer.sock` 暴露仅本用户可访问的 JSON-line 控制接口。PLLM daemon 提供：

- `GET /api/v1/expert-dataplane`；
- `POST /api/v1/expert-dataplane/actions`；
- `resize`、`prefetch`、`evict`、`evict_all`。

Prefetch、evict、resize 和 suspend 都要求 `quiesced=true`。PLLM 自动先调用 vLLM Level 0；成功后仅对可继续执行的操作 wake，失败则保持 quiesced 并进入 `ERROR`，不继续使用部分重建的层。Level 1/2 的顺序固定为 Level-0 token boundary、撤销 mappings、再释放权重；wake 后重新允许 exact miss load。

`expert_data_plane_enabled=true` 允许 daemon 发现 runtime 和接受显式操作，但 `expert_auto_resize_enabled=false` 是默认值。真实 route traces、前台 SLO 与 40 层重建验收完成前，合成 planner 不会自动触发物理 resize。

当前 40 层 resize 不是跨层原子事务。若第 N 层失败，前 N 层可能已经采用新 profile，因此 runtime 设置 `faulted=true`、`data_plane_ready=false`，恢复策略是保持 quiesced、重试同一 profile或重启 vLLM，而不是继续生成。控制命令串行执行；resize 期间 `transitioning=true`，也不会暴露 ready 状态。

## 7. 验收结果

### 已通过

- 完整回归 `63 passed`，compileall、shell syntax、CMake C++20 构建通过；
- 完整导出 40 层、20,480 objects、63,435,912,912 bytes，120 个跨层样本 checksum 全部通过；
- 128-slot Marlin EER 启动、actual Top-22 blocking miss 与真实 HTTP 生成；
- Level 1/2 后模型恢复，EER `data_plane_ready=true` 且恢复后请求 HTTP 200；
- PLLM API Level 2 在 184.5ms 回收 44.17GiB，恢复耗时 41.39s；
- 60GiB CUDA allocation 从常驻态 OOM 变为休眠态成功；
- `mlx5_0` 上 20MiB RC RDMA PUT/GET，源、remote、fetched SHA-256 一致；
- 75→71 的 4-QP volatile pool 搬运 15.859GB/5,120 个真实 expert objects：PUT 2.545s（49.85Gb/s wall），GET 到 75 本地文件 4.012s（31.62Gb/s wall）；
- wrong token、path traversal、chunk corruption、manifest-last 和 whole-epoch transaction 测试；
- 实时 API、SSE、桌面与 390px 移动控制中心渲染。

### 部分通过或失败

- 128 slots 的 exact fused Top-22 要求 `max_num_batched_tokens<=5`，首个短请求 63.94s，性能不可用于生产；
- Level 2 释放通过 3 秒目标，但本地恢复仍需约 39--42s；
- Level 0 同流冻结无 chunk、无需重连，但跨独立请求输出不 deterministic，exact token equality 未证明；
- live-state byte carrier 已测，vLLM/NemotronH 的 Mamba/KV/RNG serializer 仍未接入。

1. 真实 route trace、predictor calibration 与 128/256/full residency 对照；
2. volatile-pool v2 跨机复测、RDMA-to-slot、断链；durable path 仍需 remote disk-full；
3. Blender、游戏、NVENC 的吞吐/jank 与功耗；
