# 大模型推理暂停、状态恢复与快速加载调研

## 1. 研究问题

桌面 AI 终端需要同时回答四个不同问题：如何停止下一次 decode、如何归还 GPU/UMA 资源、如何保存正在生成的请求状态、如何快速恢复 75GB 级模型。多数框架只覆盖其中一两项。降低 batch 或并发只能降低计算，不能归还已分配权重；杀死进程能释放资源，却失去请求、KV cache 与热启动能力。

PLLM 第一版只实现 vLLM，但用同一评价表调研其他系统，防止把框架适配数量误当成研究深度。

| 系统 | 抢占粒度 | 权重释放 | 请求/KV 连续性 | 快速加载 | PLLM 采用 |
| --- | --- | --- | --- | --- | --- |
| vLLM 0.25.1 | token / scheduler | Level 1/2 | `mode=keep` + connector | fastsafetensors | 是 |
| SGLang | engine idle / RL colocate | release/resume | 通常清 KV | 权重更新路径 | 否 |
| TensorRT-LLM | inflight batch | 依赖生命周期 | host KV/offload | engine/checkpoint | 否 |
| llama.cpp/Ollama | 请求或 idle timeout | unload | 重启/重算 | mmap/本地文件 | 否 |
| CUDA checkpoint | 进程级 | 完整进程 | 理论完整 | host snapshot | 否 |

## 2. vLLM：可执行基线

vLLM Sleep Mode 的官方 Level 1 会把权重备份到 CPU 并丢弃 KV cache；Level 2 同时丢弃权重与 KV cache。在线服务必须设置 `VLLM_SERVER_DEV_MODE=1` 与 `--enable-sleep-mode`，且官方明确要求开发端点不要暴露给用户。[vLLM Sleep Mode](https://docs.vllm.ai/en/v0.17.1/features/sleep_mode/)

0.25.1 的 scheduler 进一步提供 `mode=keep/abort/wait`：

- `keep`：进入 `PAUSED_ALL`，保留运行中 request object 与 HTTP stream。
- `abort`：结束已有请求，适合作为错误恢复，而非正常桌面抢占。
- `wait`：停止接新请求但让运行请求完成；不适合游戏启动时的低延迟释放。

PLLM 的轻度路径直接使用 Level 0 + keep。深度路径先冻结 scheduler，再由版本守卫补丁执行 `reset_running_requests=True, reset_connector=False`，使请求回到 waiting queue、释放 resident block，同时保留外部 connector cache。唤醒后已落盘 Attention block 可命中，其他状态按 token 重算。

风险边界：vLLM 官方 Level 1/2 本来会忘记 KV，PLLM 补丁是实验性扩展；NemotronH 的 recurrent/Mamba state 目前没有独立 SSD 序列化器。因此“真实模型无损”必须以贪心基线逐字节比较验收，不能只看请求没有报错。

## 3. 每 token 抢占：FastServe 与 Sereno

FastServe 利用自回归推理天然存在的 token 边界做 iteration-level preemption，并主动在 GPU 与 host 间 offload/upload 中间状态。其目标是多租户服务中的延迟和吞吐，论文报告相对 vLLM 最高 6.1 倍吞吐改进。[FastServe, NSDI 2026](https://www.usenix.org/conference/nsdi26/presentation/wu-bingyang)

Sereno 面向移动设备的前后台竞争。论文观察到后台 LLM 对前台卡顿的影响高度不对称，并复用 speculative decoding 构造细粒度 yield point，在不丢推理进度的情况下动态让出内存带宽。[Sereno, OSDI 2026](https://www.usenix.org/conference/osdi26/presentation/xin)

两者给 PLLM 的启发不同：FastServe 支持“token 边界是系统级抢占点”，Sereno 支持“优化目标应该是前台 QoS，而非只看 LLM 吞吐”。PLLM 目前没有修改 speculative decoder；它先使用 vLLM scheduler 的 token step 边界，以降低比赛实现风险。后续可把 speculative draft 长度作为可调 yield quantum。

## 4. 快速加载：ServerlessLLM、MAIO 与 SparkLoad

ServerlessLLM 的关键不是简单 SSD offload，而是加载优化 checkpoint、多级存储并行加载、checkpoint locality 调度和 live migration。论文报告相对当时 serverless 系统可降低 10 到 200 倍延迟。[ServerlessLLM, OSDI 2024](https://www.usenix.org/conference/osdi24/presentation/fu)

MAIO/PPC 从文件系统 page cache 入手：用 I/O template、XPU affinity 和数据局部性控制预取与淘汰；论文报告模型加载延迟最高下降 79%。[MAIO, FAST 2026](https://www.usenix.org/conference/fast26/presentation/liu-yubo)

PLLM 不自定义 checkpoint，因为 Nemotron NVFP4 已有 17 个 safetensors shard，复制会额外占用约 75GB。SparkLoad 采取三个保守优化：

1. Level 2 只丢弃不可变权重，恢复时复用共享 shard。
2. fastsafetensors 根据 GPU 选择 GDS 或 unified copier，queue size 固定为 0，控制峰值 shard 驻留。
3. 校准脚本比较默认、multithread 与 fastsafetensors 的 p95，再选择本机最短路径。

MAIO 式 page-cache policy 仍是很好的研究扩展，但比赛版不能把 Linux 当前热 page cache 的结果写成冷盘带宽。所有 storage benchmark 都记录 `cache_state`。

## 5. KV 与混合模型状态

纯 Transformer 暂停状态主要是 request token、sampling state 与 Attention KV；NemotronH 还包含 Mamba recurrent state。HiberCache 将状态分三类：

- **持久 KV block**：FP8 原始精度，经 OffloadingConnector 写入 CPU/SSD。
- **请求控制状态**：消息、参数、已输出文本与 token 边界，由 SQLite 保存。
- **未持久混合状态**：唤醒后从已生成 token 重算；这增加恢复 TTFT，但避免不完整序列化造成静默错误。

Marlin 论文中的稀疏 token/head 思路可压缩可复用长前缀，但对正在生成请求会改变精确性边界。因此比赛中只作为独立实验组，与精确恢复结果分开报告质量、retained ratio 和恢复时间。

## 6. SGLang、TensorRT-LLM、Ollama 与 checkpoint

SGLang 的 `release_memory_occupation`/`resume_memory_occupation` 服务于 RL rollout 与 training 共卡，说明权重/缓存标签化释放是可行模式；但第一版 PLLM 不实现其适配器。[SGLang RL integration](https://docs.sglang.ai/advanced_features/sglang_for_rl.html)

TensorRT-LLM 支持 host KV cache、KV reuse 与 inflight batching，但没有直接替换 vLLM Level 0/1/2 的 PLLM 控制契约。它更适合作为以后经过独立 engine 生命周期验证的 backend。[TensorRT-LLM KV cache](https://nvidia.github.io/TensorRT-LLM/advanced/kv-cache-reuse.html)

llama.cpp 与 Ollama 的 unload/keep-alive 更接近“卸载后重载”，实现简单但请求连续性弱。CUDA process checkpoint 理论上可冻结整个 CUDA 进程，但权限、多进程、NCCL 和 CPU thread 协调的 blast radius 远大于 vLLM 内部接口。第一版不实现这些路径。

## 7. DGX Spark 的 RDMA 限制

NVIDIA 官方说明 GB10 UMA 上 GPUDirect RDMA、nvidia-peermem、DMA-BUF 与 GDRCopy 路径不工作，建议先查询 CUDA device attribute，再使用 `cudaHostAlloc` host buffer 注册 verbs MR。[DGX Spark CUDA Porting Guide](https://docs.nvidia.com/dgx/dgx-spark-porting-guide/porting/cuda.html)

因此 PLLM 的远端暖源固定为：

```text
remote memory -> ConnectX-7 RDMA WRITE -> cudaHostAlloc registered MR
              -> CPU/UMA loader -> model allocation
```

当前 RTX PRO 服务器可以探测到 GDR，也不能据此宣称 DGX Spark 支持。实验必须分列“当前 100Gbps 链路实测”和“Spark 200Gbps 规格上限”。

## 8. 结论

PLLM 的研究亮点不是重新实现一个通用 checkpoint，而是把四个已分散的问题组合成桌面闭环：Sereno 式前台 QoS 目标、FastServe 式 token 抢占、HiberCache 分层状态与 ServerlessLLM/MAIO 式加载本地性，再用 DGX Spark 的 UMA 与 RDMA 限制约束实现。第一版只支持 vLLM，使每个结论都能落到可测试 API 和故障回退上。
