# PLLM HiberFlow-EER 实验报告

> 验证日期：2026-07-18。本文只把真实硬件、真实模型或真实 RoCE
> 路径标为 **LIVE**；mock、静态检查点分析和 DGX Spark 理论值分别标注，
> 不把它们混入性能结论。

## 1. 结论摘要

本轮已在 RTX PRO 6000 Blackwell 96GB 上加载
`NVIDIA-Nemotron-3-Super-120B-A12B-NVFP4`，运行 vLLM 0.25.1、
Marlin MoE backend、FP8 KV cache 与 PLLM 128-slot Elastic Expert
Residency（EER）。真实结果如下：

- vLLM + EER 可完成生成，并可在 Level 1、Level 2 恢复后继续接受请求；
- PLLM API Level 2 在 184.5ms 内回收 44.17GiB，恢复耗时 41.39s；
- 60GiB CUDA 前台任务从模型常驻时 OOM 变为休眠后成功分配；
- 20MiB 对象真实执行 RC RDMA PUT/GET，三端 SHA-256 完全一致；
- Vue 控制中心、移动布局、SSE、vLLM 服务发现和 EER 状态均接入实时数据；
- Level 0 同一 HTTP stream 在暂停期间无新 chunk，唤醒后无需重连；
- 但跨独立请求的 greedy 输出并不 bitwise deterministic，因此尚不能声称
  “深度休眠后输出逐 token 完全等价”。

当前实现证明了数据面与资源释放闭环，但没有达到“高吞吐弹性推理”：128 个
物理槽位为了保持 fused Top-22 精确路由，必须把
`max_num_batched_tokens` 限制到 5，首个短请求耗时 63.94s。该限制是后续算法
和 kernel 设计必须解决的主要问题。

manifest 的 model fingerprint 为
`4713bfeeda725af9834232c415616656cc767040010dad7f8ad5019618185dd1`。
导出过程逐层释放临时对象，避免 private CuMem pool 累积；NVFP4 scale factor
使用等价的 scalar `amax` 路径，避免为单层创建约 1.97GiB FP32 临时张量。

### 4.2 精确路由约束

vLLM fused MoE 一次调用可能包含多个 token。若一个 batch 有 `B` 个 token，
Top-22 在最坏情况下需要 `22B` 个不同专家同时可用。128 个物理槽位要求：

```text
22B <= 128  =>  B <= 5
```

默认 profile batch 曾在单层观测到 202 个 unique experts，超出 128 slots。
当前启动脚本因此自动设置 `max_num_batched_tokens=5`。它保证 miss 时只阻塞加载
真实 Top-22，不替换路由语义，但严重损害 prefill 与 batching。这一结果说明，
论文下一版需要 layer-pipelined expert gather、分批 fused execution 或新的
pointer-table kernel，而不是仅靠 cache 策略。

### 4.3 真实推理

128-slot 服务以 `gpu_memory_utilization=0.45` 启动，稳定占用约
45.1--46.1GiB，留下约 50GiB 给桌面任务。若使用 0.85，vLLM 会把释放空间
继续分配给 KV cache，活动态占用约 83.9GiB，不符合桌面共享目标。

首个 23 prompt token + 8 completion token 请求：

| 指标 | 结果 |
| --- | ---: |
| HTTP | 200 |
| elapsed / TTFT | 63.939s |
| cumulative expert bytes loaded | 约 58.18GB |
| expert hits / misses | 13,489 / 18,787 |
| evictions | 13,667 |

Level 1 恢复后的 2-token 请求耗时 31.80s，Level 2 恢复后的 2-token 请求
耗时 36.72s。经 PLLM OpenAI 代理发送的恢复后 1-token 请求返回 HTTP 200，
耗时 33.40s。它们证明路径可用，也表明冷 expert slots 带来不可接受的恢复后
TTFT。

## 5. 显存释放与恢复：LIVE

原始数据：`results/nemotron_eer128_level1.json`、
`results/nemotron_eer128_level2.json` 和
`results/nemotron_eer128_summary.json`。

| 路径 | before | after | reclaimed | release | restore |
| --- | ---: | ---: | ---: | ---: | ---: |
| vLLM Level 1 | 44.64GiB | 1.54GiB | 43.10GiB | 16.63s | 0.639s |
| vLLM Level 2 | 45.10GiB | 1.93GiB | 43.17GiB | 0.131s | 41.92s |
| PLLM API Level 2 | 约 46.1GiB | 1.31GiB | 44.17GiB | 0.185s | 41.39s |

Level 1 会把约 25.62GiB 权重备份至 CPU，因此释放显存很慢；其优势是唤醒
只需 0.639s。Level 2 丢弃不可变权重和 KV，能迅速把 GPU 资源还给前台，
但必须从 17 个只读 shard 重载非 expert 权重，并重新填充 EER slots。

验收结果：Level 2 的“3 秒内释放至少 95% vLLM 占用”通过；Level 1 的
3 秒目标失败。恢复时间与恢复后 TTFT 仍是 SparkLoad 的首要优化对象。

`scripts/calibrate_agent.py` 已兼容 Level 2 的分段
`wake_weights + reload_weights + wake_kv` schema，并生成
`results/nemotron_calibration.json`：hibernate 130.99ms、local restore
41,916.38ms、reclaim ratio 0.9573。当前只有 1 个真实样本，该 profile 可用于
现场 Demo 的成本量级，不具备 p95 统计意义。

## 7. 请求连续性：部分通过

`mode=keep` Level 0 在同一流式 HTTP 连接上的结果：

- 第 3 个 chunk 后暂停；暂停期间 `chunks_during_pause=0`；
- 原连接唤醒后继续并正常结束，没有客户端重连；
- 独立 baseline 为 62 chars，暂停流为 64 chars，SHA-256 不相同。

控制实验又发现，即使完全不暂停，同 prompt、`temperature=0`、`seed=1234`
的独立 non-stream 与 stream 请求也分别产生 62 和 64 chars，SHA-256 不同；
短 `Say OK` 对照则相同。因此当前动态 mapping/kernel 路径本身未达到跨请求
bitwise determinism，不能把上述差异单独归因于 pause，也不能宣称“无损
恢复”已经通过。正式材料应使用“同流冻结、不重连”，将逐 token 等价列为
待解决正确性门槛。

### 8.2 Registered host staging

16MiB、32 iterations 的结果：

| allocator | MR registration | memory copy |
| --- | ---: | ---: |
| aligned | 522.705us | 132.167Gb/s |
| `cudaHostAlloc` | 422.718us | 127.338Gb/s |

这些数字是 CPU memory copy 到 registered MR 的 bandwidth，不是 100Gbps

## 11. 当前主要问题

1. **Fused Top-k 工作集上界**：128 slots 迫使 batch token cap 为 5，是当前
   63.94s 短请求的根本性能代价。
2. **恢复是两段延迟**：Level 2 资源释放很快，但非 expert 权重重载约 40s，
   首次 expert refill 再增加约 33s 用户可见延迟。
3. **runtime expert 副本占用 SSD**：20,480 对象约 60GiB，应评估 direct
   safetensors positional read 或离线重排，减少存储放大。
4. **连续性证据不足**：同流传输语义通过，采样与 kernel 的逐 token
   确定性尚未通过。
5. **FP8 KV warning**：当前未提供模型专用 calibrated FP8 KV scales，质量
   评测尚缺失。
6. **目标硬件缺口**：当前是独显服务器，不是 DGX Spark。UMA 的释放指标应以
   `MemAvailable`、PSI 和前台 admission 为主，而不是 `nvidia-smi` 显存。

## 12. DGX Spark 后续实验

在 GB10 上优先做三项，而不是重复当前所有矩阵：

1. 测量 Level 2 前后 `MemAvailable`、PSI、功耗和 Blender/游戏真实吞吐；
2. 实现按 layer microbatch 的 expert gather，解除 `B<=5`，比较 128/256/full
   slots 的 TTFT、TPOT、I/O 和前台 jank；
3. 比较 local NVMe 与第二台 Spark 的 host-staged ConnectX-7 warm source，
   分开报告网络、SSD、host-to-UMA 三段时间。

若不能解除 fused batch 容量约束，论文应把 EER 定位为“可行性与负面结果”，
主 Demo 使用快速 Level 2 释放 + 本地恢复，不声称 elastic mode 已具备生产性能。
