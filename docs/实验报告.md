# PLLM HiberFlow-EER 实验报告

## 1. 证据分级

实验结果使用四个标签，控制中心也遵循同一规则：

- **LIVE**：本机实际传感器、真实 HCA 或真实模型结果。
- **MOCK-VALIDATED**：控制协议与状态机真实执行，但 backend 不分配模型资源。
- **CONTROL-PLANE ONLY**：真实或手工资源输入产生规划投影，但没有可执行 weight data plane。
- **PENDING-GPU**：脚本与验收条件已完成，等待 GPU 空闲；不得填写估算值。

当前 GPU 上有大规模用户实验。本轮新增 expert 与 live-state carrier 数据面代码，但没有运行 pytest、CMake、vLLM、120B 模型、`cudaHostAlloc`、GDS、RDMA get/put 或 CUDA 压力程序。

## 3. 自动测试：MOCK-VALIDATED

```bash
conda activate pllm
pytest
```

数据面修改前结果：`39 passed`。覆盖：

- 新状态机、游戏 Level 2、Blender 500ms hold、yield 到 hibernate 升级。
- vLLM Level 0/1/2 `mode=keep` 调用与分阶段恢复。
- 流式输出中途冻结，暂停 120ms 无新增 chunk，唤醒后完整文本精确匹配。
- 外部无 Sleep API 服务不可控制，训练命令不被发现为 vLLM。
- SQLite token 197 冻结位置、恢复状态与 experiment schema。
- HiberCache 512MiB spec、SSD quota/LRU、扩展 API 与静态 Vue 首页。
- Reviewer/Rebuttal 双智能体的四轮下限、结构化输出解析与版本化产物写入。
- safetensors header-only expert catalog、严格 Top-22 trace schema 和 synthetic route replay。
- split-conformal marginal coverage、精确 miss fallback、分项 byte accounting 和 phase planner。
- Expert Residency API guardrail，以及独显 VRAM/统一内存两种 capacity scope。

该测试证明旧控制逻辑与同流语义，不覆盖本轮新增的数据面。`tests/test_expert_dataplane.py`、`tests/test_hiberstate.py` 和新增 API/vLLM sequencing tests 已写入但未执行。

## 4. 当前资源 Agent：LIVE + MOCK-VALIDATED

测试 daemon 读取真实 NVML，观察到受保护训练进程约 99% SM；它只向 `127.0.0.1:18000` mock vLLM 发出：

```text
sleep(level=0, mode=keep)
sleep(level=1, mode=keep)
```

一次观测的 Flask 到 mock 控制动作约 6.1ms。mock 不占 GPU，因此 `reclaimed_gb=0` 是预期结果。训练进程 PID、状态和显存不被 PLLM 修改。

守护进程保留 250ms GPU 快采样、将服务发现周期调整为 30 秒后，`pidstat -p <pid> 1 20` 的稳态平均为 0.80% CPU，RSS 67MiB，达到 <1% CPU / 200MB RAM 目标。

旧版 `results/mock_integration.json` 已升级到 schema v2，旧 `abort` 结果不作为当前方案证据。

## 5. Host-staged RDMA：LIVE

命令：

```bash
cmake -S rdma_bridge -B rdma_bridge/build -DCMAKE_BUILD_TYPE=Release
cmake --build rdma_bridge/build -j
python scripts/rdma_benchmark.py --allocator aligned --bytes 16777216 --device mlx5_0
```

结果来自 `results/rdma_bench.json`：

| 指标 | 结果 |
| --- | --- |
| HCA | mlx5_0，100Gb/sec (4X EDR) |
| host buffer | aligned，16MiB，32 iterations |
| PD/MR | 成功，remote read/write flags |
| MR 注册 | 443.205us |
| staging memcpy | 131.204Gb/s |
| GPU allocation | false |
| GPUDirect claim | false |
| 网络 RDMA | 未执行，无 peer server |

131.204Gb/s 是 CPU 内存到 registered host MR 的 copy bandwidth，不是 100Gbps 网络链路吞吐。GPU 空闲后可把 allocator 改为 `cuda-host`；双机可用后用 `--server/--peer` 单独产生网络结果。

## 7. Nemotron 实验矩阵：PENDING-GPU

### 7.1 对照组

| 组别 | 资源动作 | 恢复路径 | 请求策略 |
| --- | --- | --- | --- |
| A | kill + restart | 默认 loader | 客户端重试 |
| B | vLLM Level 1 | CPU weights | KV 重算 |
| C | Level 2 | 默认 safetensors | token 重算 |
| D | Level 2 | SparkLoad | token 重算 |
| E | Level 2 + HiberCache | SparkLoad | connector hit + tail recompute |
| F | PLLM adaptive | cost model choice | same stream |

每组至少 10 次，报告 median、p95 和置信区间；冷/热 page cache 分开。

实验完成后生成本机成本 profile：

```bash
python scripts/calibrate_agent.py results/nemotron_*.json
```

Agent 后续从 `~/.local/share/pllm/calibration.json` 读取 p95 暂停、休眠和恢复代价。

### 7.2 释放与恢复

```bash
python scripts/benchmark_sleep.py --level 1 --mode keep --wake \
  --output results/nemotron_level1.json
python scripts/benchmark_sleep.py --level 2 --mode keep --wake \
  --output results/nemotron_level2.json
```

记录：控制动作延迟、95% 资源释放时间、GPU used 或 `MemAvailable` 增量、CPU 内存增量、SSD bytes read/write、恢复时间、恢复后 TTFT、功耗积分。

验收目标：Level 2 在 3 秒内释放至少 95% vLLM 占用；该目标只有真实模型结果能判定。

### 7.3 连续生成

```bash
python scripts/benchmark_continuity.py --level 0 \
  --output results/continuity_level0.json
python scripts/benchmark_continuity.py --level 2 \
  --output results/continuity_level2.json
```

脚本先以 `temperature=0, seed=1234` 生成不中断基线，再在第 8 个 stream chunk 冻结。验收要求：暂停期间 chunk 增量为 0、stream 不重连、最终 SHA256 与基线相同。Level 2 失败时正式 Demo 自动回退 Level 0 + token state recompute，并在能力页显示降级。

### 7.4 前台 QoS

运行 Blender render 或独立 CUDA workload，比较无 vLLM 基线、模型常驻、固定 Level 2 与 PLLM adaptive。记录前台吞吐、p95 frame/render latency、GPU power 和 PLLM 检测延迟。

目标：控制动作 p95 < 500ms，释放后前台吞吐达到无后台模型基线 90% 以上。

### 7.5 Loader 与远端暖源

比较默认 loader、multithread safetensors、fastsafetensors unified/GDS。DGX Spark 只测 unified copier，不宣称 GDS/GDR。双机路径分两段记录 ConnectX-7 RDMA write 与 host staging -> UMA load；ModelExpress/NIXL 未安装时报告 fallback，不填空缺吞吐。

## 8. 报告规则

1. mock 毫秒数不能放入 75GB 权重图。
2. host memcpy 不能命名为 RDMA bandwidth。
3. RTX PRO GDR 能力不能外推到 DGX Spark。
4. page cache 未控制时标为 `uncontrolled`。
5. 真实 JSON 缺失时图表留空，不生成“示意实测值”。

## 9. Elastic Expert Residency 实验：PENDING-GPU

本节只定义协议。当前已有 runtime slot/SSD/RDMA 实现代码，但没有加载 Nemotron，也没有 expert route trace、编译结果或运行时 slot 结果。

### 9.0 Data-plane implementation：IMPLEMENTED / UNTESTED

新增代码覆盖：safetensors positional read、checksummed runtime expert package、atomic SSD tier、Marlin physical slots、actual Top-22 blocking miss、quiesced resize、Sleep suspend/resume、Unix runtime control 和 RC RDMA get/put。没有产生任何新的实验 JSON；此前 `rdma_bench.json` 不能作为新 RDMA object path 的结果。

### 9.0 Control-plane simulation：MOCK-VALIDATED

`results/expert_residency_simulation.json` 使用 3 个 synthetic requests，按 request 划分 train/calibration/test。结果只允许证明 schema、字节 accounting、exact miss fallback 和 planner transition：

| 项目 | 结果 | 口径 |
| --- | ---: | --- |
| calibration records | 960 | synthetic request 1 |
| calibration coverage | 1.0 | 非真实模型 |
| average prediction set | 263 experts | 非性能目标 |
| shifted test coverage | 0.0 | synthetic domain shift 反例 |
| 128 slots blocking miss | 45,839,238 bytes/record | synthetic accounting |
| observed-shift plan | hibernate | control-plane fallback |
| hypothetical 95% hit plan | elastic, 256 slots, 1.75 token/s | 假设输入，非测量 |

该结果刻意不出现在真实实验性能曲线中。它证明 planner 不会在明显 over-budget 的 prediction set 下强行宣称 elastic 可行。

### 9.0.1 实时控制面接入：MOCK-VALIDATED

dry-run daemon 在 `127.0.0.1:17862` 暴露 expert-residency API。当前独显上的实时压力输入得到 `elastic_resident / 128 slots/layer` 建议，并明确输出：

```json
{
  "capacity_scope": "discrete_gpu_vram",
  "evidence": "hypothetical_control_input_not_model_measurement",
  "data_plane_ready": false,
  "executable": false

```

该结果只证明实时传感器到 planner/API/UI 的连通性。daemon 不改变 controller state，不调用 vLLM weight loader，也不产生物理释放量。

### 9.1 静态检查点算术

| 项目 | 静态值 | 证据性质 |
| --- | ---: | --- |
| physical expert objects | 59.063GiB | strict `experts.<id>` header 汇总 |
| non-routed weights | 15.720GiB | safetensors header 差值 |
| layer-local expert object | 2.953MiB | 20,480 objects，等长 |
| 64 slots/layer 总权重 | 23.103GiB | 理论值，不含 runtime overhead |
| 128 slots/layer 总权重 | 30.486GiB | 理论值，不含 runtime overhead |
| all-miss expert traffic | 2.538GiB/token | 理论上界 |

这些数字来自 `results/expert_catalog.json`，不能标记为 `LIVE runtime`。完成 slot manager 后必须用 `MemAvailable`、vLLM allocation accounting 和 tensor checksum 重新测量。

### 9.2 Trace 数据集

每条 route record 至少包含：

```json
{
  "request_id": "...",
  "workload": "code|math|chat|rag|shift",
  "phase": "prefill|decode",
  "token_index": 0,
  "layer": 1,
  "actual_experts": [0, 1],
  "router_scores": [],
  "resident_before": [],
  "predicted_set": [],
  "ready_by_deadline": [],
  "miss_bytes": 0,
  "false_prefetch_bytes": 0
}
```

`actual_experts` 对 Nemotron 必须恰有 22 项。训练、calibration 和 test 按请求划分，不能把同一请求的相邻 token 分到不同 split。额外保留完全不同 prompt domain 作为 drift test。

### 9.3 Cache 与 predictor 基线

| 组别 | 预测 | 淘汰 | 动态 budget | 路由语义 |
| --- | --- | --- | --- | --- |
| E0 | 无 | LRU | 否 | exact Top-22 |
| E1 | 无 | LFU | 否 | exact Top-22 |
| E2 | route history | Least-Stale style | 否 | exact Top-22 |
| E3 | ProMoE-style | predictive | 否 | exact Top-22 |
| E4 | predictor fixed top-n | utility | 是 | exact Top-22 |
| E5 | calibrated prediction set | utility | 是 | exact Top-22 |
| E6 | E5 + I/O governor | utility | 是 | exact Top-22 |
| E7 | E6 + phase boundary | utility | 是 | exact Top-22 |

ActiveEvict-style dynamic budget 和简单阈值 `predicted_miss_bw > limit -> hibernate` 必须加入组合基线，防止复杂 controller 只击败不完整方法。

### 9.4 扫描维度与指标

- slots/layer：full、256、128、64、32；
- token rate：unlimited、10、5、2、1 token/s；
- source：NVMe only、RDMA only、best single、dual；
- foreground：none、graphics、Blender、NVENC、NVMe media；
- prompt：code、math、chat、RAG、domain shift。

核心指标：resident physical bytes、byte hit、deadline hit、prediction-set size、false-prefetch bytes、blocking I/O、TPOT、foreground throughput/jank、NVMe/RDMA/UMA bytes、功耗积分和 SSD amplification。每组至少 10 次，报告 median、p95 与置信区间。

### 9.5 正确性门槛

1. actual expert IDs 与 full-resident baseline 每层每 token 完全一致；
2. slot 中 packed weights 和全部 scales checksum 一致；
3. greedy token IDs 完全一致；
4. miss 时只能 stall，不允许 fallback 到预测 expert；
5. 无法 remap 或校验失败时必须回退 full residency/hibernate。

任何正确性门槛失败，该配置都不能进入性能图。

### 9.6 Phase-boundary 否证

扫描前台需求与 I/O 限额，比较 always-page、always-yield、always-hibernate、simple threshold 和 calibrated controller。若不存在同时优于 full residency 与 hibernate 的 elastic 区间，或复杂 controller 不优于 simple threshold，则论文应删除相应算法主张，只保留工程结论。
