# HiberFlow-EER 四轮审稿、答辩与修订记录

> 这是论文设计阶段的初始审稿记录。其容量数字已被后续 `iteration-1` header parser 纠正；请以 `docs/reviews/iteration-1/` 和最新论文为准。

## 方法说明

本记录使用 `agents/reviewer.system.md` 与 `agents/author.system.md` 定义的职责边界进行角色隔离审查。Reviewer 只评价新颖性、正确性、证据和可证伪性；Author 必须逐条选择接受、部分接受、证据反驳、转为假设或删除主张。

四轮分别聚焦：

1. 相关工作与核心创新是否只是 A+B+C；
2. DGX Spark、vLLM、NVFP4 expert slot 的工程可行性与精确性；
3. 实验、基线、指标和当前证据是否足以支撑结论；
4. 风险校准的统计边界、论文聚焦和失败条件。

最终修改已合入 `paper/HiberFlow-ACM四页稿.md` 和 `docs/PLLM项目报告.md`。本记录不是外部同行评审，也不能替代真实实验。

## 第一轮：新颖性与已有工作

### Reviewer Agent

**总体判断：4/10，Weak Reject。** 当前稿件把前台感知、expert prediction/cache、SSD/RDMA loader 和 checkpoint/restart 放在同一架构图里，但尚未说明哪个新问题不能被 ProMoE、MoE-Infinity、ActiveEvict、Sereno 与 HiberFlow 的组合直接解决。

**可信贡献**

1. 正确认识到 DGX Spark UMA 上 CPU offload 不释放物理容量。
2. 将混合 Attention/Mamba live state 与不可变权重区分。
3. 不准备修改 Top-k，存在保持模型质量的可能。

**主要问题**

`R1.1 Critical`：Expert prediction、prefetch 和 cache replacement 已有大量工作。若论文声称“首次动态专家换入换出”，该声明错误。必须逐项列出 MoE-Infinity、ProMoE、ExpertFlow、Fate、SpecMD、FlashMoE、OD-MoE 和 ActiveEvict，并删除 priority claim。

`R1.2 Critical`：ActiveEvict 已讨论动态 effective budget。作者需要说明“前台驱动 budget”是否只是换了一个 budget signal，还是产生新的约束、算法或 phase transition。

`R1.3 Major`：事务式休眠和 expert paging 看起来像两篇论文。必须给出统一目标函数，解释为什么它们不是并列功能。

`R1.4 Major`：所谓“预测不影响质量”缺少协议。若预测错误后执行其他 expert，论文就是近似推理；若等待正确 expert，需要把 stall 和 I/O 写入目标。

**可证伪条件**

如果在任何前台资源水平下，最佳策略总是 full residency 或 hibernate，而不存在 elastic Pareto 区间，则核心中间态没有价值。

**建议结论：Weak Reject。**

### Author Agent

**R1.1：接受并修改。** 删除“首次预测、预取、SSD offload、动态 budget”的所有暗示。新增相关工作矩阵，并把核心问题改为 time-varying foreground resource envelope。

**R1.2：部分接受。** 单独使用前台信号确实不构成算法。修订稿增加联合决策变量 `(expert cache budget, prediction set, token rate, hibernation)`，以及 paging 可行性不等式。主张从“新 cache policy”缩小为“外部 SLO 下 elastic-to-hibernate phase boundary”。该主张仍待实验证明。

**R1.3：接受并修改。** 状态机改为 `FULL_RESIDENT <-> ELASTIC_RESIDENT -> HIBERNATED`。Expert mode 与 hibernation 共用资源包络、I/O governor 和 chunk loader；当 miss debt 超过前台带宽约束时，transactional hibernation 是同一控制问题的退出动作。

**R1.4：接受并修改。** actual router 永远权威；预测 miss 必须加载正确 expert 并 stall。论文明确不使用预测 expert 替代品、不修改 Top-22、不进行 cache-aware rerouting。

**实际修订**

- 重写摘要、研究问题和贡献列表；
- 新增 `2.2 换容量会产生带宽债务`；
- 新增 `3.1 状态机与资源包络`、`3.4 I/O governor`；
- 新增完整相关工作与创新边界；
- 把“存在 elastic 区间”写成 H1 与否证条件，而非结论。

## 第二轮：工程可行性与精确性

### Reviewer Agent

**总体判断：5/10，Borderline。** 论文的中心问题更清楚，但 dynamic expert residency 仍停留在抽象 cache。vLLM 的 fused MoE/Marlin/ModelOpt 权重通常是固定 packed tensor，并不是可以删除的 Python objects。

**主要问题**

`R2.1 Critical`：必须说明如何真正释放物理 pages。把 logical expert 从字典删除不会释放 full packed tensor。需要 physical slot tensor、logical-to-physical remap、weight/scale object 边界和 kernel 接口。

`R2.2 Critical`：checkpoint 中每个 expert 有 weight、input scale、weight scale 等多个 tensor。只读取主 weight 会产生错误结果。必须定义原子 expert object 和 checksum。

`R2.3 Major`：CUDA graph 和 fused kernel 可能要求固定 shape。动态 cache budget 是否会触发 graph recapture、allocator fragmentation 或不支持的 pointer table？

`R2.4 Major`：论文使用“精确”一词，但没有定义检查。应比较 actual expert IDs、weight bytes、greedy tokens 和 transaction output。

`R2.5 Major`：RDMA 路径仍可能被误解成 GDR。应明确 ConnectX-7 只写 `cudaHostAlloc` staging，并把 staging-to-destination 算入时延和 UMA traffic。

**可证伪条件**

若 NVFP4 kernel 不能在固定 slot tensor上进行 logical remap，或者 remap/transform 代价超过 expert I/O overlap，则现有数据路径不可行。

**建议结论：Borderline，需大修。**

### Author Agent

**R2.1：接受并修改。** 新增 per-layer fixed physical slots 和 mapping generation。论文明确 vLLM 当前完整加载 experts，slot manager 尚未实现；只有 slot tensor 的 pages 未分配或被重用才算释放物理容量。

**R2.2：接受并修改。** 定义 expert object 包含 NVFP4 packed weight、input scale 和全部 weight scales；object 完整校验后 slot 才 ready。

**R2.3：部分接受。** 不能在尚未实现前声称 CUDA graph 热切换成功。设计选择固定 slot shape，仅改变 mapping；如 backend 不支持，则为 full/elastic 分别 capture，仍失败则回退 full/hibernate。该限制加入实现状态与失败条件。

**R2.4：接受并修改。** 新增四层正确性：actual Top-22 IDs 一致、expert object checksum 一致、greedy token 一致、fault-injected token ledger 一致。

**R2.5：接受并修改。** 所有架构图和文字统一为 `CX-7 -> ibv_reg_mr(cudaHostAlloc) -> destination`；禁止将 host memcpy 或 200Gbps line rate写成模型恢复结果。

**实际修订**

- 新增 `3.3 精确 expert slot cache`；
- 新增 expert object schema 和 ready event；
- 实现状态表把 slot manager、per-expert loader 和 Mamba transaction 标为未实现；
- 正确性实验加入 ID、checksum、token 和 epoch 四层验收；
- DGX Spark 路径统一为 no-GDR host staging。

## 第三轮：实验、基线与证据

### Reviewer Agent

**总体判断：5/10，Weak Reject。** 设计有价值，但当前没有真实 expert trace、slot cache、Nemotron release 或前台 workload 结果。论文必须避免用静态容量表暗示系统已经释放了 56GiB。

**主要问题**

`R3.1 Critical`：所有 18.51GiB、56.27GiB 数字都是 checkpoint arithmetic，不是 runtime result。必须在表题和上下文标注 allocator、workspace、KV、page cache 均未计入。

`R3.2 Critical`：缺少强基线。至少比较 full、yield、Level 2、kill/restart、LRU/LFU、MoE-Infinity/ProMoE-style、SpecMD/ActiveEvict-style 和完整系统 ablation。

`R3.3 Major`：cache hit rate 不足以证明前台收益。必须报告 byte hit、false-prefetch bytes、blocking I/O、NVMe/UMA contention、功耗和前台无后台基线。

`R3.4 Major`：Conformal predictor 没有 dataset split、coverage 或 drift protocol。

`R3.5 Major`：目标“前台达到 baseline 90%”没有样本数、p95 或置信区间。不能提前写成结果。

**可证伪条件**

若高 hit rate 伴随严重 SSD/UMA/功耗干扰，或者 resident budget 缩小后后台 miss stall 使系统频繁抖动，算法失败。

**建议结论：Weak Reject，等待真实实现。**

### Author Agent

**R3.1：接受并修改。** 所有容量数字统一标为静态算术；实现状态表与摘要明确当前没有真实释放结果。

**R3.2：接受并修改。** 增加八组基线和三项关键 ablation：without calibration、without I/O governor、without hibernation boundary。

**R3.3：接受并修改。** 指标从 activation accuracy/cache hit 扩展为 byte hit、deadline hit、false bytes、blocking I/O、前台 throughput/jank、功耗和 SSD amplification。

**R3.4：接受并修改。** 增加 train/calibration/test、跨 domain trace、rolling empirical miss 和 drift fallback。覆盖范围只称 marginal。

**R3.5：接受并修改。** `90%` 仅作为 H3 目标；要求多次运行、median/p95 和置信区间，真实值无论是否达标都进入报告。

**实际修订**

- 新增完整 `5. 评估设计`；
- 将所有性能陈述改成 H1--H5 待验证假设；
- 添加 paging collapse、能耗和前台 NVMe workload；
- 项目报告增加“已实现”和“尚未实现”两张清单；
- 删除任何从 host staging memcpy 推导网络/模型吞吐的措辞。

## 第四轮：统计边界、论文聚焦与最终裁决

### Reviewer Agent

**总体判断：6/10，Borderline / Weak Accept as design paper。** 修订稿已经诚实，但仍有统计保证和 scope 风险。

**主要问题**

`R4.1 Critical`：Split conformal 的 marginal coverage 依赖 exchangeability。autoregressive token、同一请求内 route 和 domain shift 显然相关。不能写成 per-layer/per-token 的可靠 SLA。

`R4.2 Major`：论文同时包含 foreground agent、expert predictor、cache、RDMA、transaction、Mamba 和 UI，仍容易被评价为系统堆叠。必须用一个中心优化问题贯穿，并把 Marlin/UI 放到次要位置。

`R4.3 Major`：动态 expert cache 释放容量但不减少 Top-22 FLOPs。必须明确算力让渡来自 duty cycle，而不是 expert eviction。

`R4.4 Major`：与 ActiveEvict 的差异仍应收缩。External SLO 本身不一定是算法贡献，只有 phase-boundary controller 和真实结果优于组合基线时才成立。

`R4.5 Minor`：比赛报告需要说明双智能体审稿不是外部验证，避免自我审稿被当作科学证据。

**可证伪条件**

若 rolling calibration 在真实 prompt shift 下反应晚于前台 deadline，风险校准模块没有控制价值；若 phase controller不优于最简单阈值 `miss bandwidth > limit -> hibernate`，算法复杂度缺乏必要性。

**建议结论：当前作为设计论文 Borderline；完成真实数据面和实验后可重新评估。**

### Author Agent

**R4.1：接受并修改。** 删除任何条件 coverage 暗示。公式旁明确“交换性下的有限样本边际覆盖”，并把 rolling window 定位为监控和 fallback，不声称恢复理论保证。

**R4.2：接受并修改。** 统一中心变量为前台资源包络 `E_t=(R,D,B,C,H)` 和动作 `a_t=(M_e,h,r,q)`。UI 移出论文核心；Marlin 只保留边界说明，不列入核心贡献。

**R4.3：接受并修改。** 新增明确声明：inactive expert 本来不消耗 MoE FLOPs；eviction 释放容量，算力让渡由 token bucket/decode duty cycle 实现。

**R4.4：接受并修改。** 不声称前台 signal 或 dynamic budget 本身新。创新条件改为：联合 controller 必须在同一 workload 下 Pareto 优于 ActiveEvict-style cache 加简单 hibernate threshold；否则只剩工程整合价值。

**R4.5：接受并修改。** 在本记录与项目报告标明双智能体是内部质量控制，不是外部同行评议。

**实际修订**

- 贡献收缩为资源包络、风险校准 exact residency、UMA data path 和 elastic-to-hibernate transaction；
- 增加 statistical limitation、simple-threshold baseline 和否证条件；
- 明确 memory relief 与 compute relief 的机制不同；
- 将审稿记录定位为可复现内部审查。

## 最终自查结论

### Reviewer Agent 最终意见

论文现在没有明显错误的 priority claim，也没有把静态算术或 mock 数据写成真实结果。其最有价值的问题是：在前台资源 envelope 快速变化的 UMA AI PC 上，怎样决定 exact expert residency 还能否维持，何时应转入 transactional hibernation。

当前仍不能称为已完成的顶级系统论文，因为核心 expert slot data plane、真实 route traces、Mamba transaction 和前台实验均缺失。完成这些工作后，论文是否成立取决于三条证据：

1. 存在显著的 elastic Pareto 区间；
2. 联合 controller 优于 cache policy 与简单阈值的组合；
3. 精确性、能耗和前台 QoS 同时通过，而不是只提高 cache hit。

### Author Agent 最终处理

接受该结论。最终稿保持“研究设计与系统原型”标签，不使用 Accept、SOTA 或首次等措辞。比赛 Demo 可以展示产品闭环，但论文结论只随真实数据更新。
