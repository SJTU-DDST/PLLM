# PLLM 三轮项目与审稿迭代总览

每轮先完成可测试工程增量和论文修改，再由 `Reviewer Agent` 与 `Rebuttal/Revision Agent` 进行至少三次往返。审稿人负责攻击新颖性、正确性、硬件事实与证据；作者只能接受、收缩、凭已有证据反驳或删除主张。

| 项目轮次 | 工程增量 | 论文增量 | 审稿结果 |
| --- | --- | --- | --- |
| Iteration 1 | header-only catalog、严格 trace schema、synthetic replay | 纠正 routed/non-routed 容量与 all-miss 流量 | 3 轮；删除错误容量数字，明确 synthetic 非模型证据 |
| Iteration 2 | route predictor、split conformal、exact cache simulator、phase planner | 增加边际覆盖、domain shift 与 byte accounting | 3 轮；保留 coverage=0 反例，拒绝 conditional guarantee |
| Iteration 3 | controller/API/Vue 接入、capacity scope、证据 guardrail | 增加 control-plane 实现状态与 UI/API 证据 | 3 轮；修复独显容量域和移动布局，拒绝数据面完成主张 |

当前累计 9 组 iteration-specific review/rebuttal。仓库另保留早期四轮总体审稿记录，用于比较研究方向收缩过程。

## 数据面实现后记

前三次迭代记录的是当时仅有 control plane 的状态，保留为历史审计材料，不随新实现改写。后续已增加 `.pllmex` SSD tier、vLLM ModelOpt NVFP4/Marlin physical slots 和 host-staged RC RDMA object store，并单独完成三轮源码审稿与答辩：`docs/reviews/dataplane-implementation/`。该后续代码按显存约束尚未构建或运行，不能用前三轮的 39 项回归替代数据面证据。

自动编排实现位于 `pllm/peer_review.py` 与 `scripts/run_peer_review.py`。它要求至少三轮、保存 manuscript snapshots，且默认不覆盖主论文。当前 GPU 忙碌，因此本轮采用同一角色协议对代码、测试、API 输出和论文做离线审计，没有启动本地大模型审稿服务。
