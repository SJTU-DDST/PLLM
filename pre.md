PLLM：面向 DGX Spark 的可暂停大模型推理框架
各位老师好，我们是来自上海交通大学 DDST 实验室的"注意力算得队"。我们的项目是 PLLM——一个面向 DGX Spark 的前台感知、可暂停大模型推理框架。它的核心是精确专家驻留、分级休眠与资源感知调度算法。
我们的团队来自 上海交通大学 DDST 实验室，研究方向包括分布式内存，存储系统，AI系统加速等

一、背景与问题
DGX Spark 这类桌面 AI 工作站有一个现实矛盾：前台生产任务与后台大模型推理并重，但功耗、显存和统一内存资源非常有限。

后台侧，基于 Nemotron 120B FP4 模型的各类推理任务要占用约 75 GiB 显存，GPU 计算压力也较大；而前台侧，用户发起的图像生成、渲染、游戏、视频编解码又需要低延迟拿到 GPU 资源。用户突然切回前台时，后台模型该怎么办？

现有方案里，主流大模型和Agent一旦终止推理流程，就会破坏生成进度，必须从头 recompute；如果后台模型持续驻留内存，前台任务会 OOM 或卡顿；不加处理完全卸载到 SSD，reload weights 则需要很长时间。

由此提炼出四个核心问题：如何感知前台压力并按需释放资源？如何决断高价值参数（活跃 MoE 与 KV cache）的驻留与低价值参数的驱逐？如何以 token 为粒度判定暂停边界、保持推理连续性？如何从 SSD 或远端存储快速恢复？

二、解决方案
PLLM的目标是构建一个前台感知的弹性推理运行时。从用户视角看，PLLM是一个综合监测显存容量、算力、功耗和前台任务状态的Agent，动态地暂停和恢复，保持连接不中断的同时释放显存，在前台体验与后台连续性之间动态平衡。从工程性框架看，PLLM构建分层模型数据路径，支持内存、硬盘、RDMA 远端多级存储与恢复，整个系统通过统一控制平面展示资源、进行决策。学术创新点看，PLLM会优先恢复高价值专家和活跃状态，缩短恢复关键路径，同时保证了模型准确性。

首先介绍与模型有关的 insight 和算法框架设计。我们选用了Nemotron 120B NVFP4，占用显存75G左右。其中专家约59G，非路由稠密部分16G。 Moe的架构是40层*512个专家，每个token激活22个专家。由此引申出弹性专家驻留算法：Prefill阶段强制 512专家全部驻留，Decode阶段引入Hot Set Expert策略，采样过去 256-token local window，生成下一窗口的较热expert提前预取或保留在内存中，其余驱逐到ssd或远端。为保证正确性，真正激活的 Top-22 始终执行，如果预测 miss 则重新加载加载正确专家。专家打包则引入了Sequential pack 策略，将待恢复专家文件首尾拼成连续 pack 并记录对象位置，减少元数据查询和零散 I/O。被卸载的专家的位置可以是本机nvme，也可以rdma到远端存储池。最后通过一个状态机判断前端系统压力，进行动态迁移。


三、项目框架
PLLM 运行时分为三层。

使用者与前台环境：接入 OpenAI 客户端；用 Qt 和 Vue 分别实现桌面悬浮窗和网页控制面板；GNOME 插件以 250 微秒间隔把系统数据源源不断传到控制中心。

计算节点：分两个进程。PLLM 进程里，状态机负责模型休眠、冻结与恢复的统一仲裁；Monitor 和 Planner 每 250ms 融合前台信号进行专家驻留规划。vLLM 推理进程里，EER Runtime 管理 expert slot 状态与 miss 加载。

远端预注册存储池作为 volatile expert pool，C++ RDMA 服务通过 TCP 控制 + RC Verbs 提供高速回填。

一次推理请求的主路径：请求进入 → PLLM Proxy 准入转发 → Prefill 强制 512 slots/层 Full 驻留 → Decode Router 产出权威 Top-22 → 判断 Expert 是否在 Slot，命中直接执行，未命中则从本地 NVMe 或远端 RDMA 校验加载、推理等待，再输出 token。

四、MoE 驻留与驱逐算法
驱逐策略分三级。Level 0 / FULL：Dense/Router/Shared 与全部 MoE Expert、KV Cache 全驻留，请求流正常运行。Level 0 → ELASTIC：对 MoE 专家做热驻留 / 冷驱逐——热专家保留在 GPU slot，冷专家迁到 SSD/RDMA 副本；KV Cache 仍全保留；向前台部分释放容量。Level 0 → Level 2（UMA 默认）：Dense/Router/Shared、全部 MoE Expert、大部分 KV Cache 全部驱逐，仅保留请求流与恢复信息；KV 落 SSD KV + 清单，模型落共享模型目录，向前台大部分释放容量。

接下来请看我们的项目 demo。由于模型加载时间较长，视频部分片段已加速。

首先启动三个模块：启用 GNOME 插件识别前台窗口；启动桌面悬浮窗；启动 PLLM daemon，在 17860 端口监听前台应用负载。这个时候agent就启动完毕了。我们可以加载 vLLM 在后台进行推理，为 380 expert slots 每层分配空间、开启 Sleep Mode 与 HiberCache；

等待两分钟出现加载完成以后，桌面弹窗和网页 dashboard 都会识别到 vllm 服务的存在。可以看到在没有其他前台负载的情况下，所有专家和kv都在显存中。

此时可以运行 start_llm_load，开启推理流，可以从控制面板和命令行看到推理输出。

现在我们给 GPU 一个较轻的前台负载，比如打开 Blender 演示工程。

此时可以在悬浮窗看到看到：PLLM 检测到 Blender 是前台应用了。我们触发GPU渲染，需要算力，但是显存尚且撑得住。因此模型生成暂停，状态机进入 YIELDING、HIBERNATED，前端实时显示释放动作与容量，Blender 获得更多 GPU 资源。

渲染完成或按 Esc 停止后，PLLM 等待系统空闲，进入 RESTORING 再回到 ACTIVE，模型恢复输出，完成不带专家驱逐的快速恢复。

对于比较重的前台负载，PLLM将执行L2的深层释放。为了模拟这个场景，我们给 PLLM 下发前台优先的策略。

同样触发渲染，PLLM还是会先进入YIELD状态，在持续一段时间以后会进行 yield escalate。观察右边的悬浮窗，显存在较短的时间内被释放，同时观察 dashboard，提示模型资源已经释放，且看dashboard的下半部分，热expert被放到了local nvme中。


最后我们等待一段时间，前台任务不再占用资源，系统将刚刚驱逐的专家等重新加载回到显存。

此外，dashboard 和悬浮窗都可以选择，是倾向于前台资源优先（优先深层次释放），还是保证后台不被驱逐，当然所有的驱逐也可以手动触发，不一一列举了。


五、实验结果
我们在 RTX PRO 6000 96GB 上实测 vLLM 0.25.1 的 Nemotron 120B NVFP4 模型。选择性专家驱逐策略相比将中间状态完整 offload 到 SSD，关键路径缩短 59%；两者均释放约 62.6–62.7 GiB GPU 空间。

此外，我们测试了热点专家 GPU 驻留数量与性能关系：K=256 + miss 重新加载从 1.0476 秒降到 0.7902 秒，加速 24.57%；仅唤醒 K=256 个热点专家时 H2D 最快加速 1.33×，首批搬运量降至 41.1 GiB。

消融实验方面，我们证明了 Sequential pack 和 Hot set 设计的有效性。二者结合既减少恢复对象和数据量，又提高 SSD 读取连续性，延迟由 56.5 秒降至 21.9 秒，获得 2.59 倍加速。

结语
一句话总结：PLLM 是一个 负载感知大模型推理框架，他让 DGX Spark 能跑 120B 大模型，又能随时让出资源给前台任务，在 token 边界安全暂停、按热度精确驱逐、从分层存储快速恢复。谢谢各位评委老师，敬请批评指正！
